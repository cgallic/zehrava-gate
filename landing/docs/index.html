Zehrava Gate Docs

Gate sits between your AI agents and production systems. Every write — email sends, CRM imports, database updates, file publishes — goes through Gate before it lands. Policies are YAML. Evaluation is deterministic. Every decision is audited.

Quickstart

Start the Gate server with npx — no install required for the server itself:

npx zehrava-gate --port 4000

Or install globally:

npm install -g zehrava-gate
zehrava-gate --port 4000

Server options:

FlagDefaultDescription
--port4000HTTP port to listen on
--data-dir./dataDirectory for SQLite database and stored payloads
--policy-dir./policiesDirectory containing your YAML policy files

✓ Self-hosting: Gate stores all data locally in SQLite. Nothing leaves your infrastructure. No cloud dependency, no API keys for the server itself.

Register an agent

Before an agent can propose actions, it needs to be registered. Registration returns an API key — store this securely. It cannot be retrieved again.

curl -X POST http://localhost:4000/v1/agents/register \
  -H "Content-Type: application/json" \
  -d '{"name": "my-crm-agent", "riskTier": "standard"}'
{
  "agentId": "agt_a1b2c3d4e5f6",
  "apiKey":  "gate_sk_••••••••••••••••••••",
  "name":    "my-crm-agent",
  "riskTier": "standard"
}

⚠ Save the API key immediately. Gate stores only a hash — if you lose it, register a new agent.

The riskTier field is stored for auditing. Current values: standard, elevated, restricted. Future policy enforcement will gate actions by tier.

Your first proposal

Create a policy file first. Drop it in your --policy-dir:

# policies/support-reply.yaml
id: support-reply
destinations: [zendesk.reply, intercom.reply]
auto_approve_under: 1
block_if_terms:
  - "refund guaranteed"
  - "legal action"
expiry_minutes: 30

Then propose an action from your agent:

const { Gate } = require('zehrava-gate')
const gate = new Gate({ endpoint: 'http://localhost:4000', apiKey: 'gate_sk_...' })

const p = await gate.propose({
  payload:     'Thank you — your issue has been resolved.',
  destination: 'zendesk.reply',
  policy:      'support-reply',
  recordCount: 1
})

console.log(p.status) // "approved" | "blocked" | "pending_approval"
console.log(p.proposalId)
console.log(p.blockReason) // set if status === "blocked"

Gate evaluates the policy immediately and returns one of three statuses:

  • approved — passed all rules, within auto-approve threshold. Safe to act.
  • blocked — violated a policy rule. Do not proceed. Check blockReason.
  • pending_approval — within policy but requires human review. Poll or wait for webhook.

Policy overview

Policies are YAML files stored in your --policy-dir. Gate loads them on startup and caches them. Each policy has an id that your agent references in the propose call.

Evaluation runs top-to-bottom through these checks — the first match wins:

1

Policy exists?

If the referenced policy ID has no YAML file, Gate returns blocked immediately.

2

Destination allowlist

If destinations is set and the proposal's destination isn't in the list, Gate blocks it.

3

Payload type check

If allowed_types is set and the payload's file extension isn't in the list, Gate blocks it.

4

Term scan

If block_if_terms is set, Gate scans the payload content for each term. Matching is normalized — strips punctuation, collapses whitespace, maps leet substitutions. First match blocks the proposal.

5

require_approval: always

If set, every proposal goes to pending regardless of other rules.

6

Record count thresholds

auto_approve_under: if recordCount ≤ threshold, approve automatically.
require_approval_over: if recordCount > threshold, send to pending.

7

Default

If no rule matched, Gate returns pending_approval. Safe default — when in doubt, ask a human.

All policy fields

FieldTypeDescription
id * string Unique identifier. Must match the filename (without .yaml). Referenced in propose calls.
destinations string[] Allowlist of destination identifiers. Proposals to any other destination are blocked. Omit to allow any destination.
allowed_types string[] Allowlist of file extensions without dot (e.g. [csv, json, txt]). Only applies when payload is a file path.
block_if_terms string[] List of terms to scan the payload for. Matching is normalized — case-insensitive, punctuation-stripped, leet-decoded. First match blocks with reason: "Payload contains blocked term: <term>"
require_approval string Set to "always" to require human approval on every proposal regardless of other rules. Recommended for high-risk destinations.
auto_approve_under number If recordCount is ≤ this value, the proposal is auto-approved. Useful for single-record operations that don't need review.
require_approval_over number If recordCount exceeds this value, the proposal goes to pending. Use alongside auto_approve_under to define a safe/unsafe threshold.
expiry_minutes number How long a pending proposal can wait for approval before it expires. Expired proposals cannot be approved. Default: 60.

Policy examples

Support reply — auto-approve safe messages, block sensitive terms

id: support-reply
destinations: [zendesk.reply, intercom.reply, freshdesk.reply]
allowed_types: [text, json]
block_if_terms:
  - "refund guaranteed"
  - "legal action"
  - "sue"
auto_approve_under: 1   # single reply → auto-approve
expiry_minutes: 30

CRM import — auto-approve small batches, review large ones

id: crm-low-risk
destinations: [salesforce.import, hubspot.contacts]
allowed_types: [csv, json]
auto_approve_under: 100        # ≤100 records → auto
require_approval_over: 100     # >100 records → human review
expiry_minutes: 60

Finance — always require human approval

id: finance-high-risk
destinations: [stripe.refund, quickbooks.journal, netsuite.payment]
require_approval: always
expiry_minutes: 15              # short window — stale approvals are dangerous

Internal blog publish — scoped destination, always approve

id: blog-publish
destinations: [blog.publish, cms.publish]
require_approval: always
expiry_minutes: 120

API — Register agent

POST /v1/agents/register
Register a new agent and receive an API key. No authentication required — this is the bootstrap endpoint.

Request body

FieldTypeDescription
name *stringHuman-readable agent name. Appears in audit logs and the dashboard.
riskTierstringstandard | elevated | restricted. Default: standard.
 
{
  "agentId":  "agt_a1b2c3d4",
  "apiKey":   "gate_sk_••••••••••••",
  "name":     "my-agent",
  "riskTier": "standard"
}

API — Propose

POST /v1/propose
Submit an agent action for policy evaluation. Returns immediately with a status.

Authentication: Authorization: Bearer <apiKey>

FieldTypeDescription
destination *stringTarget system identifier (e.g. zendesk.reply). Must match your policy's destinations allowlist.
policy *stringPolicy ID to evaluate against (matches YAML filename without extension).
payloadstringThe content being proposed. Text content, JSON string, or file path (detected by extension). Term scanning only applies to text content.
recordCountnumberNumber of records in a batch operation. Used for threshold evaluation (auto_approve_under, require_approval_over).
expiresInstringExpiry duration: 30m, 1h, 2h. Default: 1h. Overrides policy expiry_minutes.
on_behalf_ofstringAgent ID of the orchestrating agent in multi-agent pipelines. Logged in the audit trail for accountability.
metadataobjectArbitrary key/value pairs. Stored with the proposal and included in audit events.
 
{
  "proposalId": "prop_a1b2c3d4e5f6",
  "status":     "approved",          // approved | blocked | pending_approval
  "blockReason": null,               // set if blocked
  "expiresAt":  "2026-03-08T12:00:00.000Z"
}

API — Approve

POST /v1/approve
Approve a pending proposal. Returns a one-time delivery token.

Authentication: Authorization: Bearer <apiKey>

FieldTypeDescription
proposalId *stringThe proposal ID returned from /v1/propose.
 
{ "status": "approved", "deliveryToken": "dlv_••••••••••••" }

Blocked proposals cannot be approved — the endpoint returns 409.

API — Reject

POST /v1/reject
Reject a pending proposal. Sets status to blocked.

Authentication: Authorization: Bearer <apiKey>

FieldTypeDescription
proposalId *stringProposal to reject.
reasonstringRejection reason. Stored in audit trail.

API — Deliver

POST /v1/deliver
Get a one-time download URL for an approved proposal's payload. The URL can only be used once — subsequent calls return 410 Gone.

Authentication: Authorization: Bearer <apiKey>

{
  "url":           "https://your-gate-server/v1/download/dlv_••••",
  "deliveryToken": "dlv_••••••••••••",
  "expiresAt":     "2026-03-08T12:00:00.000Z"
}

API — List proposals

GET /v1/proposals?status=pending_approval&limit=50
List proposals with optional status filter. Used by the dashboard.

Authentication: Authorization: Bearer <apiKey>

ParamTypeDescription
statusstringFilter by: pending_approval, approved, blocked, delivered, expired. Omit for all.
limitnumberMax results. Default: 50.

API — Audit trail

GET /v1/audit/:proposalId
Full audit trail for a proposal — every event in order: proposed, policy_checked, approved/blocked, delivered, expired.

Authentication: Authorization: Bearer <apiKey>

[
  { "eventType": "proposed",       "actor": "my-crm-agent", "createdAt": "..." },
  { "eventType": "policy_checked", "actor": "system",       "metadata": { "result": "approved" } },
  { "eventType": "auto_approved",  "actor": "system",       "createdAt": "..." },
  { "eventType": "delivered",      "actor": "system",       "createdAt": "..." }
]

SDK — Methods

gate.propose(opts)

Mirrors the POST /v1/propose endpoint. All fields from the API are supported.

const p = await gate.propose({
  payload:      './leads.csv',
  destination:  'salesforce.import',
  policy:       'crm-low-risk',
  recordCount:  847,
  on_behalf_of: 'agt_orchestrator_id',  // optional
  expiresIn:    '2h'                      // optional
})
// p.proposalId, p.status, p.blockReason, p.expiresAt

gate.approve({ proposalId })

await gate.approve({ proposalId: p.proposalId })

gate.reject({ proposalId, reason })

await gate.reject({ proposalId: p.proposalId, reason: 'Flagged for review' })

gate.deliver({ proposalId })

const { url } = await gate.deliver({ proposalId: p.proposalId })
// url is a one-time download link — use it once, then it's gone

gate.verify({ proposalId })

Fetch full proposal details including audit trail.

const detail = await gate.verify({ proposalId: p.proposalId })
// detail.status, detail.auditTrail, detail.blockReason, ...

Python SDK

Install the Python SDK — zero dependencies, works with Python 3.8+. Supports LangGraph, CrewAI, AutoGen, and any custom agent.

pip install zehrava-gate
from zehrava_gate import Gate, GateError

gate = Gate(
    endpoint="http://localhost:4000",
    api_key="gate_sk_..."
)

p = gate.propose(
    payload="Thank you — your issue is resolved.",
    destination="zendesk.reply",
    policy="support-reply",
    record_count=1
)

# p["status"] → "approved" | "blocked" | "pending_approval"
if p["status"] == "blocked":
    raise GateError(p["blockReason"])

LangGraph integration

from langgraph.graph import StateGraph
from zehrava_gate import Gate, GateError

gate = Gate(endpoint="http://localhost:4000", api_key="gate_sk_...")

def send_crm_import(state):
    p = gate.propose(
        payload=state["csv_path"],
        destination="salesforce.import",
        policy="crm-low-risk",
        record_count=state["record_count"],
        on_behalf_of=state.get("orchestrator_id")
    )
    if p["status"] == "blocked":
        return {**state, "error": p["blockReason"]}
    if p["status"] == "pending_approval":
        return {**state, "proposal_id": p["proposalId"], "waiting": True}
    return {**state, "sent": True}

Webhooks

Instead of polling for approval status, register a webhook URL. Gate fires a POST to your URL when the proposal is approved or rejected.

// JavaScript
const p = await gate.propose({ ... })

await gate.registerWebhook({   // JS SDK — coming in next release
  proposalId: p.proposalId,
  url: 'https://your-app.com/gate-callback',
  secret: 'your-secret'           // sent as X-Gate-Secret header
})
# Python
gate.register_webhook(
    proposal_id=p["proposalId"],
    url="https://your-app.com/gate-callback",
    secret="your-secret"
)

Gate fires once — on the first approve or reject event. The webhook payload:

{
  "proposalId": "prop_a1b2c3",
  "event":      "approved",        // or "rejected"
  "deliveryToken": "dlv_••••",     // only on approved
  "approver":   "agent-name",
  "firedAt":    "2026-03-08T11:30:00.000Z"
}

Webhooks are in-memory. If the Gate server restarts before the webhook fires, the registration is lost. Persistent webhook storage is on the roadmap. For now, use webhooks for short-lived proposals.

Dashboard

The dashboard at /dashboard on your Gate server shows every pending proposal. No code required to review and approve — it's the interface for humans in the loop.

For each proposal you can see:

  • Agent name, destination, policy, record count
  • Time remaining before expiry
  • Payload hint (type + size — not full content by default)
  • Full audit trail of events so far

Try it at zehrava.com/dashboard →

Approval is one click. The reviewer doesn't need an API key or code access — just the dashboard URL. Gate handles the auth internally for dashboard actions.

Audit trail

Every proposal generates an immutable sequence of audit events, written server-side. Agents have no API route to write or modify audit events.

Event types in order:

  • proposed — agent submitted the proposal
  • policy_checked — Gate evaluated the policy
  • auto_approved — passed all rules within auto-approve threshold
  • blocked — violated a policy rule (includes reason)
  • approved — human approved in dashboard (includes approver name)
  • rejected — human rejected (includes reason)
  • delivery_url_issued — one-time download URL created
  • delivered — payload downloaded (consumed)
  • expired — proposal passed its expiry window

Proposal lifecycle

propose
  ↓
policy evaluation
  ├── blocked        → terminal. Cannot be approved.
  ├── approved       → delivery token issued. One-time download available.
  └── pending        → waits in dashboard for human action
                          ├── approved → delivery token issued
                          ├── rejected → terminal (status: blocked)
                          └── expired  → terminal. Cannot be approved.

Fail closed

If Gate is unreachable, your agent must halt — not proceed.

Gate is middleware. If the HTTP call to /v1/propose fails with a network error, the safe behavior is to treat it as a block and not execute the write. Proceeding silently bypasses Gate entirely.

try {
  const p = await gate.propose({ ... })
  if (p.status === 'blocked') throw new Error(p.blockReason)
  if (p.status === 'pending_approval') return // wait — do not proceed
  // status === 'approved' → safe to act
} catch (err) {
  // Gate unreachable or returned error — fail closed
  console.error('Gate check failed — halting write', err.message)
  throw err // do NOT proceed
}

Multi-agent pipelines

In pipelines where one agent orchestrates others, use on_behalf_of to record the full chain of accountability in the audit trail:

// Leaf agent calling Gate on behalf of the orchestrator
await gate.propose({
  payload:      data,
  destination:  'salesforce.import',
  policy:       'crm-low-risk',
  on_behalf_of: 'agt_orchestrator_abc123'
})

The audit trail will record both the calling agent and the orchestrating agent. Without on_behalf_of, only the leaf agent appears — making the orchestrator's role invisible in the audit.

✓ Best practice: Wire Gate at the orchestrator level, not just leaf agents. If the orchestrator calls Gate before dispatching to sub-agents, the accountability chain is cleaner.
