Autonomous Agents Guardrails Policy Enforcement For agent builders

Autonomous agents act fast. Gate decides what they're allowed to act on.

Autonomous agents are useful because they don't wait for instructions. That's also the problem. An agent that can decide and act without a checkpoint between those two things will eventually act on a bad decision. Gate is that checkpoint — between every decision and every production write.

How to add guardrails to autonomous AI agents

Guardrails for autonomous agents work by inserting a policy checkpoint between the agent's decision and the action it wants to execute. The agent submits an intent — what it wants to do, with what payload, to what destination. Gate evaluates that intent against a YAML policy and returns one of three decisions: approved (proceed), blocked (stop), or pending (wait for a human).

Effective guardrails must be deterministic: the same policy always produces the same result for the same input. LLM-based moderation does not meet this bar. Gate uses YAML policies that specify explicit rules: field validation, value bounds, rate limits per agent, and environment-specific approval thresholds. Self-hosted so agent payloads never leave your infrastructure. Fail-closed by design — if Gate is unreachable, actions do not proceed. Add Gate to any agent: npm install zehrava-gate, register your agent, drop a policy file, point the agent at the SDK.

The failure mode Gate addresses

Autonomous agents fail in a specific pattern: they make a locally-reasonable decision that is globally wrong. The model didn't malfunction. The tool call was syntactically valid. The action just shouldn't have happened — because the amount was too high, the scope was too broad, or the timing was wrong.

Gate catches these failures before they execute. Not by second-guessing the model's reasoning — by checking the action against explicit rules you define.

Low-risk actions (auto-approve)

  • Read operations
  • Internal status updates
  • Single-record writes below threshold
  • Staging/dev environment writes

High-risk actions (require approval)

  • Bulk writes above N records
  • Outbound emails
  • Financial transactions
  • Production API calls to third parties

The split is your call. Define it in YAML. Gate enforces it.

How it works

1
Agent submits intent Agent calls gate.propose() with payload, destination, and policy ID. This takes < 5ms on auto-approve.
2
Gate evaluates the policy Field checks, rate limits, environment thresholds, sensitive terms — all evaluated deterministically from your YAML file.
3
Three possible outcomes approved — agent proceeds immediately. blocked — action halted, reason returned. pending_approval — human reviews on dashboard.
4
Execution reported back After running an approved action, agent reports success or failure. Full lifecycle — proposed to succeeded — recorded in audit log.
5
Webhook fires On any terminal event — approved, blocked, rejected — Gate fires your registered webhook. Route to Slack, PagerDuty, or your own handler.

SDK: three lines to add a checkpoint

const { Gate } = require('zehrava-gate');
const gate = new Gate({ endpoint: 'http://localhost:3001', apiKey: process.env.GATE_API_KEY });

const intent = await gate.propose({
  payload:     JSON.stringify({ records: bulkData, target: 'salesforce' }),
  destination: 'salesforce.api',
  policy:      'crm-low-risk',
  recordCount: bulkData.length,
});

if (intent.status === 'approved') {
  await runSync(bulkData);
} else if (intent.status === 'blocked') {
  throw new Error(`Gate blocked: ${intent.blockReason}`);
} else {
  // pending_approval — poll or use X-Gate-Wait: true to hold connection
  await waitForApproval(intent.intentId);
}

Patterns for common agent architectures

Loop agents (plan → act → observe → repeat)

Call gate.propose() at the act step. On pending_approval, pause the loop and wait. Resume on approval using the execution token. The agent's observe step runs after Gate confirms execution.

Multi-agent pipelines (orchestrator + workers)

Register each agent separately. Use on_behalf_of when an orchestrator submits on behalf of a worker — Gate tracks both identities in the audit log. Set per-agent rate limits to prevent runaway workers.

LangChain / LangGraph agents

Use GateTool from @zehrava/langchain-gate to wrap any LangChain Tool. Use gateNode + gateRouteAfter to branch a StateGraph based on Gate's decision. No changes to your tool logic.

Existing agents (no code changes)

Route outbound HTTP through the Gate proxy: HTTP_PROXY=http://localhost:4001. Gate intercepts all calls, submits intents automatically, and blocks or holds based on your destinations.yaml. Set GATE_TLS_INTERCEPT=true to intercept HTTPS.

What Gate is honest about

Gate does not protect against every failure mode. Write your expectations down before deploying:

  • Gate catches what your policies describe. An action your YAML doesn't cover gets evaluated against the default policy — pending approval by default, configurable.
  • Gate does not interpret natural language. Field checks are pattern matches and value comparisons. They catch "amount: 999999", not "please send a large charge".
  • Gate protects against mistakes, not adversarial agents. A model that modifies the policy files it runs under is out of scope. Gate's trust boundary is the policy directory.
  • Fail-closed is real. If Gate is unreachable, the SDK throws and the action does not proceed. Build your agent to handle this gracefully.
