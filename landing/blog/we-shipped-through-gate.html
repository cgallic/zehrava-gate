We Shipped Our Blog Launch Through Gate. Here's What the Approval Queue Looked Like.

✓ This post was proposal prop_d4a2b9 · approved · delivered once · audit trail at /dashboard

Three blog posts. Four AI agents. One approval queue. Zero posts went live until a human said so.

That's the short version. Here's what the AI agent content publishing workflow actually looked like when we ran our own content launch through Zehrava Gate.

What We Did

We needed three SEO posts for the Gate launch. Rather than write them manually, we spun up four sub-agents simultaneously:

  • agent-seo: Pulled 115 keywords across 7 clusters — agent governance, AI commit checkpoint, policy-as-code, audit trail gaps, CRM agent writes, and two more. Saved the full map to a shared scratchpad file the other agents could reference.
  • agent-blog1, agent-blog2, agent-blog3: Each took one post brief and wrote in parallel. No waiting. Total wall-clock time for three drafts: under four minutes.

After the drafts landed, a fifth agent — agent-publisher — ran an SEO pass using the keyword map, built the styled HTML pages, and submitted all three to Gate using the blog-publish policy.

The posts:

  • Why AI Agents Need a Commit Checkpoint
  • 847 Records, No Audit Trail: The AI Agent Write Problem
  • Policy-as-YAML: Agent Governance That Scales Across Teams

None of them published automatically. They sat in the approval queue until we reviewed them.

The blog-publish Policy

Gate is policy-driven. Every agent action goes through a policy that defines what's allowed, what requires approval, and what gets blocked. Here's the policy the publisher agent used:

id: blog-publish
destinations: [blog.publish]
require_approval: always
expiry_minutes: 1440
notes: "All blog content must be reviewed before publishing to zehrava.com/blog"

The key: require_approval: always. The agent can propose. It cannot deliver without a human in the loop. gate.propose() creates the proposal and returns an ID. gate.deliver() only executes after an approver signs off — and it's one-time. Once it fires, the proposal closes.

Same System, Any Write Type

Gate doesn't know these were blog posts. It doesn't have a blog mode. The same dashboard where we approved these proposals is where we approve Salesforce record imports, support email drafts, and finance exports. The payload is different every time. The pipeline is identical:

  1. Agent calls gate.propose()
  2. Policy evaluates destination and payload
  3. Proposal lands in the queue
  4. Human reviews and approves
  5. gate.deliver() fires — once

A blog post going through Gate looks the same to the system as a Stripe refund request going through Gate. Same approval UX. Same audit log. Same policy enforcement. The AI agent content publishing workflow is one instance of a broader pattern.

That's what "framework-agnostic" means in practice. Gate sits between your agents and production, regardless of what your agents are doing.

Governance as Infrastructure

Most teams bolt on approval flows after something goes wrong. An agent writes to the wrong CRM record. An email goes to the full list. A post publishes before it's ready. Then they add a check. The problem: every new agent action needs its own check. You end up with a patchwork of custom guards — each one fragile, none of them consistent.

Gate flips the model. Define policies once. Every agent action that matches a destination goes through the same checkpoint automatically. The agent doesn't need to know your approval requirements — it calls gate.propose(). Gate handles the rest.

Governance becomes infrastructure. Not a feature you add. Not a wrapper you bolt on. The checkpoint is in the path, and everything passes through it.
